## init

Adopt the current app in place.

```bash showLineNumbers
npx better-convex init
npx better-convex init --yes
npx better-convex --backend concave init --yes
npx better-convex init --cwd apps/web
npx better-convex init --env-file .env.agent --yes
```

Behavior:

- `better-convex init` writes the Better Convex baseline only:
  - `convex/functions/schema.ts`
  - `convex/functions/http.ts`
  - `convex/lib/crpc.ts`
  - `convex/lib/get-env.ts`
  - `concave.json`
- On backend `convex`, `better-convex init` runs `convex init` before the first Better Convex codegen pass.
- Convex deployment-targeting flags such as `--prod`, `--preview-name`, `--deployment-name`, and `--env-file` are forwarded to `convex init`.
- feature bundles stay out of `init`. Layer them with:
  - `better-convex add auth`
  - `better-convex add ratelimit`
  - `better-convex add resend`
- `--cwd` picks the target directory for in-place adoption.
- `--json` includes `codegen` (`generated` or `stubbed`) and `convexBootstrap` (`existing`, `created`, or `missing`) for automation.
- backend resolves from `--backend`, then `meta["better-convex"].backend`, then `convex`

Options:

| Flag | Description |
|------|-------------|
| `--backend <convex\|concave>` | Backend CLI to drive |
| `--cwd <cwd>` | Target directory |
| `--prod` | Forward to `convex init` |
| `--preview-name <name>` | Forward to `convex init` |
| `--deployment-name <name>` | Forward to `convex init` |
| `--env-file <path>` | Forward to `convex init` |
| `--yes` | Deterministic non-interactive mode |
| `--json` | Machine-readable command output |

## create

Create a fresh starter app, then layer Better Convex on top.

```bash showLineNumbers
npx better-convex create -t next --yes
npx better-convex create -t vite --yes
npx better-convex create -t next --yes --cwd apps --name web
npx better-convex create -t next --yes --env-file .env.agent
```

Behavior:

- `better-convex create -t next` creates a shadcn Next.js app, then patches in:
  - `app/convex/page.tsx`
  - `components/providers.tsx`
  - `lib/convex/*`
  - `convex/functions/messages.ts`
  - `convex/functions/generated/messages.runtime.ts`
  - the Better Convex baseline files
- `better-convex create -t vite` creates a Vite app, then adds:
  - `components/providers.tsx`
  - `lib/convex/query-client.ts`
  - `lib/convex/crpc.tsx`
  - `lib/convex/convex-provider.tsx`
  - `.env.local`
  - the Better Convex baseline files
- Template creation preserves shadcn-owned shell files and only patches the Better Convex seams.
- Template creation detects `src/` vs root app layouts and writes the client scaffold into the matching tree. Mixed `src` + root layouts fail fast instead of guessing.
- On backend `convex`, `better-convex create` runs `convex init` before the first Better Convex codegen pass.
- Convex deployment-targeting flags such as `--prod`, `--preview-name`, `--deployment-name`, and `--env-file` are forwarded to `convex init`.
- After scaffold creation, layer optional bundles with:
  - `better-convex add auth`
  - `better-convex add ratelimit`
  - `better-convex add resend`

Options:

| Flag | Description |
|------|-------------|
| `-t, --template <template>` | Project template to create. Supported values: `next`, `vite`. |
| `--backend <convex\|concave>` | Backend CLI to drive |
| `--cwd <cwd>` | Parent directory for the new app |
| `--name <name>` | Name for a new target directory |
| `--prod` | Forward to `convex init` |
| `--preview-name <name>` | Forward to `convex init` |
| `--deployment-name <name>` | Forward to `convex init` |
| `--env-file <path>` | Forward to `convex init` |
| `--yes` | Deterministic non-interactive mode |
| `--defaults` | Use default shadcn answers without enabling the rest of `--yes` |
| `--json` | Machine-readable command output |

## add

Scaffold plugin files and register the plugin in `schema.ts` idempotently.

```bash showLineNumbers
npx better-convex add <plugin>
```

If `<plugin>` is omitted in a TTY, the CLI prompts you to select one interactively.

In interactive TTY mode, `add` also prompts for scaffold files:

- File list = all templates exposed by the plugin manifest.
- Default selection:
  - lockfile mappings for that plugin if present
  - otherwise templates from the resolved preset
- You can add/remove files before scaffolding.
- Empty selection is rejected.

In non-interactive mode (`--yes` or no TTY), `add` uses the same resolved default template set directly.

By default, scaffold files are created under:

- `<functionsDir>/plugins/<plugin>.ts` for Convex function exports.
- `convex/<paths.lib>/plugins/<plugin>/...` for helper modules.

Write behavior for selected files:

- Missing file: created.
- Existing identical file: skipped.
- Existing changed file:
  - `--overwrite`: overwrite directly.
  - Interactive TTY without `--overwrite`: prompt per file.
  - Non-interactive without `--overwrite`: skip and report.

Options:

| Flag | Description |
|------|-------------|
| `--yes` | Deterministic non-interactive mode |
| `--json` | Machine-readable command output |
| `--dry-run` | Show the full install plan without writing files |
| `--diff [path]` | Show unified diffs for planned file changes (implies `--dry-run`) |
| `--view [path]` | Show planned file contents (implies `--dry-run`) |
| `--overwrite` | Overwrite existing changed files without prompt |
| `--no-codegen` | Skip automatic codegen after add |
| `--preset <name>` | Plugin preset override (plugin-specific allowed values) |

If the Better Convex baseline is missing, `add` bootstraps it first. The preview surface covers the full repo delta:

- scaffold files
- env helper bootstrap
- `concave.json`
- `convex.json` when `functionsDir` is `convex/functions`
- `<functionsDir>/schema.ts`
- `<functionsDir>/http.ts`
- `<paths.lib>/crpc.ts`
- `<functionsDir>/plugins.lock.json`
- dependency install status
- codegen and `hooks.postAdd`
- env reminders

Preview output uses semantic comparisons for `.ts`, `.tsx`, `.js`, `.jsx`, and `.json`, so formatter-only changes are treated as unchanged.

`--diff [path]` and `--view [path]` accept workspace-relative output paths. Matching is exact path first, then substring, then suffix.

Examples:

```bash showLineNumbers
npx better-convex add <plugin> --preset <preset>
npx better-convex add resend --diff convex/plugins/resend.ts
npx better-convex add resend --view convex/schema.ts
```

Plugin helper scaffold location is configurable via `meta["better-convex"].paths.lib` in `concave.json` (no CLI path flag).  
Plugin function modules scaffold to `<functionsDir>/plugins/<plugin>.ts`.

## view

Inspect a plugin install plan without writing files:

```bash showLineNumbers
npx better-convex view <plugin>
```

If `<plugin>` is omitted in a TTY, the CLI prompts you to choose a plugin.

Template selection is deterministic:

- Default source is `plugins.lock.json` template mappings.
- If lockfile mappings are missing, fallback is the resolved preset template set.
- Passing `--preset` forces that preset template set.

`view` prints the resolved preset, selection source, docs link, planned files, and planned operations. `--json` returns the same plan payload shape as `add --dry-run --json`.

Options:

| Flag | Description |
|------|-------------|
| `--json` | Machine-readable install plan output |
| `--preset <name>` | Plugin preset override |

## info

Inspect project and installed plugin state:

```bash showLineNumbers
npx better-convex info
```

`info` reports:

- resolved backend
- detected config paths
- installed `better-convex` and backend package versions
- schema and lockfile presence
- installed plugins
- schema/lockfile mismatches
- missing package dependencies
- scaffold drift per plugin

Options:

| Flag | Description |
|------|-------------|
| `--json` | Machine-readable inspection output |

## docs

Show local and public docs links for CLI topics and plugins:

```bash showLineNumbers
npx better-convex docs resend cli
```

Supported topics include:

- `cli`
- `plugins`
- `auth`
- `orm`
- `migrations`
- `resend`
- `ratelimit`

Options:

| Flag | Description |
|------|-------------|
| `--json` | Machine-readable docs link output |
